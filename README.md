# Charlotte — Voyage Archive

An interactive web archive of the family circumnavigation aboard *Charlotte* (2001–2005). Map, log entries, logbook scans, and trip photos — all driven by plain JSON data files.

**Design doc:** see `plan.md` for full architecture and roadmap.

---

## Quick start (view the site)

The site loads data with `fetch()`, so you need a local web server — opening `index.html` directly will not work.

```bash
cd LogBook
python -m http.server 8000
```

Open **http://localhost:8000**

The original single-file prototype is kept as `charlotte_logbook.html` for reference.

## Folder map — where everything goes

```
LogBook/
├── Scans/                      ← Original PDF batches as received (archive, don't edit)
├── transcripts/
│   └── drafts/                 ← Optional scratch notes while transcribing
├── data/
│   ├── manifest.json           ← Site metadata + list of voyage legs
│   └── leg-XX-….json           ← Transcribed stops & entries (source of truth)
├── images/
│   ├── scans/                  ← Cropped logbook images, one per entry (by date)
│   └── photos/
│       ├── intake/             ← Drop new trip photos here (Phase 3)
│       ├── web/                ← Web-sized copies (generated by script, Phase 3)
│       └── thumbs/             ← Thumbnails (generated by script, Phase 3)
└── images/originals/           ← Full-res photo backups (gitignored, local only)
```

| What | Where | Naming |
|------|-------|--------|
| Incoming scan PDFs | `Scans/` | Any clear name, e.g. `Pg8-Pg14.pdf` |
| Working transcription notes | `transcripts/drafts/` | Optional `.txt` / `.md` while you work |
| **Final transcribed text** | `data/leg-XX-….json` | Structured JSON — this is what the site reads |
| Cropped logbook page images | `images/scans/` | `YYYY-MM-DD.jpg` (one per log entry) |
| Trip photo originals | `images/originals/` | Keep locally; not in git |
| Trip photos for the site | `images/photos/web/` + `thumbs/` | Generated later by resize script |

## Workflow: adding new logbook scans

When you receive a new batch of scanned pages:

### 1. Archive the PDF

Save the file unchanged in `Scans/`.

### 2. Transcribe into JSON

Open the appropriate leg file in `data/` (e.g. `leg-01-florida-bahamas.json`). For each log entry, add or update an object in the stop's `entries` array:

```json
{
  "date": "2002-02-15",
  "date_display": "Saturday, February 15, 2002",
  "scan": "images/scans/2002-02-15.jpg",
  "body": "Transcribed text, verbatim. Uncertain words in [brackets].",
  "conditions": "Wind: NE 15 knots. Distance: 42 miles."
}
```

**Field guide:**

| Field | Required | Notes |
|-------|----------|-------|
| `date` | Yes | ISO format `YYYY-MM-DD` — used for sorting and image naming |
| `date_display` | Yes | As written in the logbook |
| `scan` | Yes | Path to cropped image: `images/scans/YYYY-MM-DD.jpg` |
| `body` | Yes | Full narrative text |
| `conditions` | No | Wind/distance summary line, or `null` |

Use `transcripts/drafts/` for rough notes if helpful, but **the JSON file is the source of truth** the website displays.

### 3. Crop and save entry images

When a logbook page has **multiple entries**, crop each entry separately. Save as:

```
images/scans/2002-02-15.jpg
```

The filename must match the `scan` path in JSON. JPG or PNG both work — keep the extension consistent.

Until the image exists, the site shows a labeled placeholder automatically.

### 4. Add or update stops

Each **stop** is an anchorage or location on the map:

```json
{
  "id": "stop-11",
  "n": 11,
  "name": "Port Antonio, Jamaica",
  "passage": "Windward Passage to Jamaica",
  "lat": 18.179,
  "lng": -76.452,
  "photo_album": null,
  "photos": [],
  "entries": [ … ]
}
```

| Field | Notes |
|-------|-------|
| `passage` | Section header in the log UI, e.g. `"Florida–Bahamas Passage"`. Change when the narrative moves to a new passage. |
| `lat` / `lng` | Map pin position |
| `photo_album` | URL to external full-res album (Amazon Photos, etc.) — `null` until photos exist |
| `photos` | Array of in-repo thumbnails — empty until Phase 3 |

When you add stops to a new region, either extend an existing leg file or create a new one and register it in `data/manifest.json` with `"status": "in-progress"`.

### 5. Preview

```bash
python -m http.server 8000
```

Refresh the browser after saving JSON or adding images.

## Workflow: trip photos (Phase 3 — not yet active)

No trip photos exist for the early voyage. When you reach the Pacific and photos become available:

### Storage strategy

- **Thumbnails + web-sized copies** live in the git repo (`images/photos/thumbs/`, `images/photos/web/`) — fast to load on the site.
- **Full-resolution originals** stay in `images/originals/` (gitignored) or on **Amazon Photos** (or similar).
- Per stop, set `photo_album` to the external share link for full-quality viewing.

### Per-stop photo references

```json
"photo_album": "https://www.amazon.com/photos/share/…",
"photos": [
  {
    "file": "images/photos/web/2003-04-12_001.jpg",
    "thumb": "images/photos/thumbs/2003-04-12_001.jpg",
    "date": "2003-04-12",
    "caption": "Sunset at anchor"
  }
]
```

When either `photos` or `photo_album` is set, a **Trip photos** link appears on that stop in the log. Thumbnails open in an on-site gallery; the external link opens the full album.

### Intake folders (future)

Organize undated scanned prints by stop before processing:

```
images/photos/intake/stop-14/
images/photos/intake/stop-14/IMG_0001.jpg
```

Helper scripts (`scripts/resize-photos.py`, `scripts/match-photos.py`) will be added in Phase 3.

## Leg files vs. passages

- **Leg files** (`leg-01-florida-bahamas.json`, etc.) are organizational — one file per major voyage segment. They keep JSON manageable as the archive grows. The UI does **not** show "Leg 1 / Leg 2" headers.
- **Passages** (`passage` field on each stop) are what readers see — thematic labels like `"Florida–Bahamas Passage"` or `"The Exumas to Georgetown"`. Set `passage` on each stop; a new sticky header appears whenever the value changes.

---

## manifest.json

Lists all legs. Only legs with `"status": "in-progress"` or `"status": "complete"` are loaded by the site. Planned legs stay in the manifest for reference but are skipped until you start transcribing them.

```json
{ "id": "leg-02", "name": "Caribbean & Panama", "file": "leg-02-caribbean-panama.json", "status": "planned" }
```

Change to `"in-progress"` when you create that leg's JSON file.

## Hosting (when ready)

Push to GitHub and enable **GitHub Pages** on the repo. The site is static — no server required in production.

⚠️ Google Drive does not serve HTML as web pages. Use GitHub Pages, Netlify, or Cloudflare Pages.

⚠️ Full-resolution photos may exceed GitHub's ~1 GB repo limit. Plan on thumbnails in git + external albums for full quality (see Phase 3 above).

## Transcription conventions

- Transcribe verbatim; preserve the original voice.
- Uncertain readings: `[brackets]`.
- Keep wind/distance summaries in `conditions`, separate from `body`.
- Entries within a stop are chronological.
